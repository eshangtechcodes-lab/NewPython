## 五、2026-03-05 工作记录

### 15:00 - 15:15 | 原 API 性能瓶颈分析

对原 C# API 53 个接口进行性能实测（`scripts/perf_check_old_api.py`），定位慢接口根因。

**测试结果：**

| 分类 | 数量 | 占比 |
|------|------|------|
| ⏰ 超时(>60s) | 9 | 17% |
| 🔴 极慢(>10s) | 2 | 4% |
| 🟡 慢(5-10s) | 3 | 6% |
| ✅ 正常(<2s) | 37 | 69% |

**根因定位（以 GetSPRegionList 7条数据耗时6.2s为例）：**

- **不是 SQL 慢，是 WCF 中间层架构开销**
- C# API 不直连数据库，通过 WCF (`wsHttpBinding` + `Message` + 证书认证) 调用后端服务
- 每次请求链路：`HTTP → Controller(new Transaction) → WCF调用 → 后端Service → Oracle → 回传`
- WCF 占总耗时 ~70%，Transaction 初始化 ~20%，DataTable 序列化 ~10%

### 15:15 - 15:43 | 基线缓存采集 + 快速对比方案

**问题**：旧 API 太慢导致每次对比测试都需 60+ 分钟。

**解决方案**：两步走策略

| 步骤 | 脚本 | 用途 | 耗时 |
|------|------|------|------|
| Step 1 | `scripts/baseline_collect.py` | 并发采集旧API → 缓存到 JSON | ~15分钟（只需跑一次） |
| Step 2 | `scripts/compare_cached.py` | 读缓存 + 调新API → 对比 | ~3分钟（后续每次） |

**基线采集结果（2线程并发，90s超时）：**
- 124个接口，总耗时 908s
- 成功: 42  超时: 6  HTTP错误: 76
- 结果保存: `scripts/test_results/baseline_cache.json`

### 15:43 | 缓存对比测试结果
