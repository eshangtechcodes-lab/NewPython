# 项目建议与总体规划

> 项目：EShangApi C# → Python FastAPI 兼容式迁移
>
> 创建时间：2026-03-09
>
> 目的：沉淀对当前项目的判断、风险、建议、收尾计划，以及后续接口知识总结的统一入口。

## 1. 项目定位

本项目不是重新设计一套新 API，而是将原有手写 C# Web API 按“行为兼容”的原则平移到 Python FastAPI。

核心要求不是代码风格一致，而是以下内容对齐：

- 路由路径一致
- HTTP 方法一致
- 入参格式一致
- Header 读取逻辑一致
- SQL 查询逻辑一致
- 默认分页、默认排序一致
- 删除语义一致（软删除 / 真删除）
- 新旧接口返回字段、类型、值尽量一致

## 2. 当前状态判断

基于 2026-03-09 对工作流、SOP、实施计划及仓库代码的阅读，当前项目已经进入收尾阶段。

- 主体迁移方法已经成型，工作流比较稳定
- 绝大多数 Controller 已经完成迁移

当前最明确的剩余业务模块是 `ShopVideoController`，包含 16 个接口，属于最后的功能性收尾。

### 2.3 当前阶段结论

项目已不处于“方案探索期”，而处于“最后收尾 + 工具修正 + 文档统一 + 知识沉淀”阶段。

## 3. 目前暴露出的主要问题

### 3.1 进度文档口径不完全一致

不同文档中的接口数、剩余数、阶段说明存在历史残留，说明当前进度统计仍有人工维护误差。

直接影响：

- 容易误判真实剩余工作量
- 后续交接时容易产生理解偏差
- 计划、执行、验收的口径可能不一致

### 3.2 对比脚本能力弱于 SOP 要求

当前 `scripts/compare_api.py` 更像临时检查脚本，还不够支撑正式验收。

当前不足包括：

- 默认只针对单一接口
- 默认只发空参数
- 默认只按 POST 处理
- 主要对比首条数据
- 没有标准化支持 3 组参数验证
- 没有系统处理 Header、忽略字段、GET/POST 差异

### 3.3 迁移脚本仍保留“无参全量迁移”能力

虽然 SOP 明确禁止无参全量执行，但脚本本身仍允许这样做，存在误操作风险。

### 3.4 配置管理尚未完全收口

应用本身已有 `config.py` 和环境变量配置思路，但部分脚本仍写死数据库连接信息。

直接影响：

- 安全性差
- 环境切换麻烦
- 后续自动化执行不稳定

### 3.5 主入口注册方式偏人工

`main.py` 中已累积大量 `include_router(...)`，后续继续扩展、重构或交接时维护成本较高。

## 4. 建议优先级

### P0：先保证收尾正确

1. 完成 `ShopVideoController` 剩余 16 个接口迁移
2. 每完成一批严格执行基准采集、迁表、对比验证、文档回填
3. 不要为了速度跳过 SOP 中的对比验证和问题记录

### P1：先修工具，再扩大交付可信度

1. 增强 `compare_api.py`
2. 收紧 `server_migrate.py` 的误操作入口
3. 统一脚本配置来源到 `config.py` 或 `.env`

### P2：统一文档真源

1. 明确哪个文档是“进度真源”
2. 明确哪个文档是“建议与规划真源”
3. 明确哪个文档是“接口知识沉淀真源”

### P3：为长期维护补结构

1. 建立模块级接口知识总结
2. 建立表依赖视图
3. 建立统一验收清单
4. 视情况把路由注册改成集中注册机制

- 降低后续维护和交接成本

- 清理历史统计口径冲突
- 将旧进度表标记为“历史记录”或“已废弃”
- 保留一个总计划文档作为最终进度真源

### 阶段 D：补接口知识资产

目标：

- 让后续新增迁移、排错、验收不再依赖单次人工阅读
