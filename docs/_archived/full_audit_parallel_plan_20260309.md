# 全量接口一致性审计并行方案

审计日期：2026-03-09

## 目标

这次不是抽查，而是对“当前已经迁移好的接口”做全量比对。

目标拆成两层：

1. 契约层全量比对  
   对比路由、HTTP 方法、参数名、Header、响应包结构。
2. 逻辑层全量比对  
   对比 Python Router/Service 与原 C# Controller/Helper 的业务逻辑是否完全一致，包括：
   - SQL 语义
   - 默认筛选条件
   - 分页
   - 排序
   - 删除语义
   - 汇总/OtherData
   - Token/Header 注入
   - 外部接口调用
   - 是否为占位/简化版/空实现

## 当前基线

机器基线文件已经生成：

- `docs/full_audit_baseline_20260309.json`

生成脚本：

- `scripts/export_full_audit_baseline.py`

这份 JSON 的用途不是直接下结论，而是给多窗口统一口径：

- 哪些模块已经迁移
- 哪些路由在 Python 运行时存在
- 哪些路由在原 C# 中存在
- 哪些已经出现方法/参数/响应包漂移

注意：

- 这份 JSON 只是“路由级基线清单”，不是最终审计结论。
- 每个窗口不能停留在路由对比，必须继续下钻到实际接口逻辑。
- 最终“完全一致/不一致”的判断，必须以 Python Router/Service 与 C# Controller/Helper 的实际逻辑比对为准。

## 审计深度

每个窗口对每个已迁移接口，必须按下面的深度执行，不允许只看路由名或方法名：

1. 先做路由基线确认
   - 路由是否存在
   - HTTP 方法是否一致
   - 参数名是否一致
   - 响应包结构是否一致
2. 再做接口实现比对
   - Python Router 的入参与 Header 处理
   - Python Service 的业务分支
   - Python Service 的 SQL 语义
   - 原 C# Controller 的参数注入与权限逻辑
   - 原 C# Helper 的核心查询/更新/删除逻辑
3. 最后做一致性判定
   - 只有“契约层 + 逻辑层”都一致，才能标记为 `完全一致`
   - 只要出现简化版、空实现、替代实现、返回空列表、直接成功、缺少关键分支，均不能标记为 `完全一致`

## 什么叫“实际接口逻辑比对”

下面这些都属于必须检查的“逻辑”，不是可选项：

- 查询 SQL 是否等价
- 默认过滤条件是否等价
- 是否读取 ProvinceCode / ServerpartCodes / ServerpartShopIds / Token / UserPattern
- 分页、排序、树结构、OtherData、汇总字段是否一致
- 新增/更新时是否补齐关联字段
- 删除是软删还是真删
- 是否有审批流、日志、历史备份、状态流转
- 外部接口调用是否被本地空实现替代
- 是否存在 `return []` / `return True` / “简化版”“复用版”“占位版”

## 每个窗口必须遵守的统一规则

1. 不允许抽样，负责模块内所有已迁移路由都必须分类。
2. 每个路由只能归入以下状态之一：
   - `完全一致`
   - `契约不一致`
   - `逻辑不一致`
   - `占位实现`
   - `Python 多出`
   - `Python 缺失`
3. 每条结论必须给出证据：
   - Python 文件
   - C# 文件
   - 关键函数或关键语句
4. “可能不一致”不能直接下结论，必须写成：
   - `已确认不一致`
   - `待联调验证`
5. 发现空实现、简化实现、直接 `return []/True`，一律按 `占位实现` 处理，不能算“一致”。
6. 不允许只做 Router 对比，必须下钻到 Service 和原 C# Helper。
7. 如果只确认了“路由存在”，但没有确认实际逻辑，则该接口不能标为 `完全一致`。

## 每个窗口的输出格式

每个窗口都写到自己的结果文件，严格使用下面的结构。

### 1. 模块总表

表头固定：

| 模块 | 已迁移路由数 | 完全一致 | 契约不一致 | 逻辑不一致 | 占位实现 | Python 多出 | Python 缺失 | 备注 |

### 2. 路由级问题清单

每条问题必须包含：

- 路由名
- 问题级别：`P0 / P1 / P2`
- 问题类型：`契约 / 逻辑 / 占位 / 缺失 / 多出`
- Python 证据
- C# 证据
- 差异说明
- 修复建议

### 3. 模块整改建议

每个模块至少输出：

- 第一优先级修什么
- 可以批量修什么
- 必须人工联调验证什么

### 4. 风险与阻塞

例如：

- 原 C# Helper 找不到
- SQL 逻辑过深，需要联调才能确认
- 外部支付/文件上传无法在静态审计中闭环

## 结果汇总口径

所有窗口结果回来后，由当前窗口统一汇总成两份内容：

1. 总审计报告
   - 各模块真实完成度
   - 各模块一致性状态
   - P0/P1/P2 问题总表
2. 实施计划
   - 第一阶段：先修契约层
   - 第二阶段：补齐占位实现
   - 第三阶段：补缺失路由
   - 第四阶段：联调与回归验证

## 建议执行顺序

建议先并行出 4 份窗口结果，再由当前窗口汇总。

推荐顺序：

1. 先运行窗口 2 和窗口 3  
   原因：Finance / MobilePay / Audit / Analysis / Verification / Sales 的“占位实现”风险最高。
2. 再运行窗口 4  
   原因：Picture / Video 的路由集合偏差最大，容易先形成 P0 清单。
3. 最后运行窗口 1  
   原因：BaseInfo / Merchants / Contract 的问题更多是深逻辑和老兼容细节。
