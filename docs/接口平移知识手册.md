## 一、技术栈差异对照

### 1.1 运行时环境

| 项目 | C# 旧系统 | Python 新系统 |
|------|----------|--------------|
| 语言 | C# .NET Framework | Python 3.11 |
| Web 框架 | ASP.NET Web API | FastAPI + Uvicorn |
| 数据库驱动 | Oracle.ManagedDataAccess | dmPython（达梦数据库） |
| ORM/数据访问 | DataTable / DataRow | 原生 SQL + dict 结果集 |
| 部署 | IIS | Uvicorn（开发模式）/ Gunicorn（生产） |
| 配置 | Web.config | Python 配置文件 |

### 1.2 语言级差异（高频踩坑点）

| C# 写法 | Python 等效 | ⚠️ 易踩坑点 |
|---------|------------|------------|
| `value.TryParseToDecimal()` | `float(value) if value is not None else 0.0` | Python `float(None)` 会抛异常 |
| `value.ToString()` | `str(value)` | `str(40.0)` → `"40.0"` 而非 `"40"`；需用 `int(v) if v == int(v) else v` |
| `DataTable.Compute("max(X)", filter)` | 自行遍历 list 过滤取 max | 注意 None 值处理 |
| `string.IsNullOrWhiteSpace(x)` | `not x or not x.strip()` | Python 空字符串 `""` 是 falsy |
| `Math.Round(x, 2)` | `round(x, 2)` | Python 银行家舍入 vs C# 四舍五入 |
| `null` | `None` | C# 返回 `null` 的字段，Python 有时填充了 `0.0`，需对齐 |
| `0 == false` (仅 bool) | `0` 是 falsy | **重大坑**: `str(score or "")` 当 `score=0` 时返回空串 `""`，应改为 `str(score) if score is not None else ""` |
| `DataRow["X"].ToString()` 对 DBNull 返回 `""` | `str(row.get("X") or "")` | 注意 `or` 会吞掉 `0`、`0.0` 等 falsy 值 |
| `ref` 参数修改 | 直接赋值返回 | C# `GetSubType(ref BusinessTrade, ...)` 会修改传入变量 |

### 1.3 数据结构映射

| C# 类型 | Python 对应 | 备注 |
|---------|------------|------|
| `DataTable` / `DataRow` | `list[dict]` | 数据库查询结果 |
| `List<T>` | `list` | |
| `Dictionary<K,V>` | `dict` | |
| `decimal` / `decimal?` | `float` / `None` | Python 无原生 decimal 区分 |
| `NestingModel<T>` | `{"node": {...}, "children": [...]}` | 树形嵌套结构 |
| `HolidayINCDetailModel` | `{"curYearData":, "lYearData":, "increaseData":, ...}` | 增长分析模型 |

### 2.2 ⚠️ 同名字段不同含义

> [!CAUTION]
> `BUSINESS_TRADE` 在不同表中**含义完全不同**，这是最容易踩的坑之一！

| 表 | 字段 | 实际值 | 含义 |
|----|------|--------|------|
| `T_SERVERPARTSHOP` | `BUSINESS_TRADE` | `244`, `245`, ... | **门店实际业态 ID**（如 244=商超零售） |
| `T_BUSINESSWARNING` | `BUSINESS_TRADE` | 多为 `NULL` 或 `1000` | 固化数据中的业态标记，**不是**具体业态 ID |

**影响**: 如果用 `T_BUSINESSWARNING.BUSINESS_TRADE` 做业态过滤，几乎不会命中正确数据。正确做法是查 `T_SERVERPARTSHOP`。

### 2.3 BusinessTradeType 映射规则

C# 中 `BUSINESS_TYPE` / `BUSINESSTRADETYPE` 与业态分类的对应关系：

| BusinessTradeType | 含义 | SQL 条件 |
|-------------------|------|----------|
| 1 | 自营餐饮 | `BUSINESS_TYPE = 4000 AND SHOPTRADE = 2` |
| 2 | 自营非餐饮 | `BUSINESS_TYPE = 4000 AND SHOPTRADE <> 2` |
| 3 | 非自营（联营） | `BUSINESS_TYPE <> 4000` |
| 1+2 | 全部自营 | `BUSINESS_TYPE = 4000` |

## 三、已踩坑清单

> 按问题严重性排序，每个坑标注影响范围，便于快速排查。

### 坑 #1 — SQL 过滤 vs 内存过滤混淆 🔴

**严重性**: 高 — 导致数据完全错误  

**场景**: 某些接口中 `BusinessTrade` 参数的过滤位置

| | C# 实现 | Python 常见错误 |
|---|---|---|
| **在哪过滤** | 内存遍历时按 `T_SERVERPARTSHOP.BUSINESS_TRADE` 判断 | 放在 SQL `WHERE` 中按 `T_BUSINESSWARNING.BUSINESS_TRADE` 过滤 |
| **结果** | 正确只返回匹配门店 | 返回全量数据（因为 BW 表的 BUSINESS_TRADE 几乎都是 NULL） |

**涉及接口**: `GetShopSABFIList`

**排查方法**: 对比 C# 源码，确认 BusinessTrade 是放在 SQL 里还是在 `foreach` 循环中用 `dtShop` 过滤。

---

### 坑 #2 — INNER JOIN 隐式排除自营门店 🔴

**严重性**: 高 — 自营门店数据静默丢失  

**现象**: 查 `T_PROFITCONTRIBUTE` 时用 `INNER JOIN T_BUSINESSPROJECT`，自营门店（无 `BUSINESSPROJECT_ID`）被排除。

**正确做法**: 
```sql
-- ❌ 错误: 排除了无 BUSINESSPROJECT_ID 的行
FROM T_PROFITCONTRIBUTE A JOIN T_BUSINESSPROJECT B ON A.BUSINESSPROJECT_ID = B.BUSINESSPROJECT_ID

-- ✅ 正确: 保留自营门店
FROM T_PROFITCONTRIBUTE A LEFT JOIN T_BUSINESSPROJECT B ON A.BUSINESSPROJECT_ID = B.BUSINESSPROJECT_ID
```

**涉及接口**: `GetShopSABFIList`

### 坑 #3 — GetSubType 递归展开缺失 🟡

**严重性**: 中 — 传子级业态 ID 时过滤范围不完整  

**业务逻辑**: C# 中 `GetSubType(ref BusinessTrade, ..., "2000")` 会在 `T_AUTOSTATISTICS`（`TYPE=2000`）中递归查找所有子类型 ID，然后将原始 ID 和所有子 ID 一起用于过滤。

**Python 实现模板**:
```python
# GetSubType 递归展开业态子类型
if shopTrade:
    _bt_ids = [s.strip() for s in shopTrade.split(",") if s.strip()]
    _expanded = set(_bt_ids)
    _sub_rows = db.execute_query(
        "SELECT AUTOSTATISTICS_ID, AUTOSTATISTICS_PID FROM T_AUTOSTATISTICS WHERE AUTOSTATISTICS_TYPE = 2000"
    ) or []
    _changed = True
    while _changed:
        _changed = False
        for sr in _sub_rows:
            sid = str(sr.get("AUTOSTATISTICS_ID") or "")
            pid = str(sr.get("AUTOSTATISTICS_PID") or "")
            if pid in _expanded and sid not in _expanded:
                _expanded.add(sid)
                _changed = True
    expanded_shop_trade = ",".join(_expanded)
```

**涉及接口**: `GetShopSABFIList`, `GetMonthINCAnalysis`, `GetMonthINCAnalysisSummary`

**不涉及**: `GetBusinessTradeRevenue`, `GetBusinessTradeLevel`（C# 也没有 GetSubType）

**业务逻辑**: `T_PROFITCONTRIBUTE` 表中 `CALCULATE_SELFSHOP` 区分自营(1)和联营(2)两套评分数据

**C# 原始 SQL**: 

**Python 对应**: 
```python
selfshop_val = 2 if BusinessTradeType == "3" else 1
```

### 坑 #5 — 利润/成本汇总来源错误 🟡

**严重性**: 中 — 带 BusinessTrade 过滤时数值不对  

| | C# 实现 | Python 错误做法 |

### 坑 #6 — Python `or` 吞掉 falsy 值 🟡

**严重性**: 中 — 0 值显示为空  

```python
# ❌ 错误: score=0 时 (0 or "") 返回 ""
value = str(score or "")

# ✅ 正确: 只有 None 时才为空
value = str(score) if score is not None else ""

# ✅ 正确（整数格式对齐 C# ToString）:
if score is not None:
    sv = float(score)
    value = str(int(sv)) if sv == int(sv) else str(sv)
else:
    value = ""
```

### 坑 #9 — 日期格式不一致 🟡

**严重性**: 中 — 前端展示和数据解析受影响

**现象**: C# 返回 `2025/4/16 2:30:40`（不补零），Python 返回 `2025-04-16 02:30:40`（ISO 格式补零）

| 字段 | C# 格式 | Python 常见输出 |
|------|---------|---------------|
| 日期时间 | `yyyy/M/d H:mm:ss` | `YYYY-MM-DD HH:MM:SS` |
| 纯日期 | `yyyy/M/d` | `YYYY-MM-DD` |

**正确做法**:
```python
# 对齐 C# yyyy/M/d 不补零格式
def format_date_csharp(dt_obj):
    if not dt_obj: return ""
    if isinstance(dt_obj, datetime):
        return f"{dt_obj.year}/{dt_obj.month}/{dt_obj.day}"
    # 字符串转换去除前导零
    parts = str(dt_obj).split("-")
    return f"{parts[0]}/{int(parts[1])}/{int(parts[2])}" if len(parts) == 3 else str(dt_obj)
```

**涉及接口**: `GetMEETINGList`（Examine 模块）、多个带日期字段的接口

### 坑 #10 — null vs 空串行为差异 🟡

**严重性**: 中 — 前端判断逻辑可能出错

**现象**: C# 对 `DBNull` 返回空串 `""`，Python 对 `None` 值直接返回 `null`

| 字段 | C# 返回 | Python 返回 |
|------|---------|------------|
| `SERVERPART_REGION` | `""` | `null` |
| 数值字段（无数据） | `null` | `0.0` |

**排查要点**: 
- 字符串字段：C# `DBNull → ""` → Python 应该用 `str(val) if val is not None else ""`
- 数值字段：C# `null` → Python 不要填充默认值 `0.0`，应保持 `None`

**涉及接口**: `GetMEETINGList`、`GetMonthINCAnalysis` 等

---

### 坑 #11 — 查询异常静默成功（假通过） 🔴

**严重性**: 高 — 掩盖真实错误

**现象**: 接口查不到数据（表不存在/环境问题），但 `try/except` 捕获后直接返回空成功结果 `Result_Code=100, data=[]`

```python
# ❌ 错误: 吞掉异常，返回空成功
try:
    rows = db.execute_query(sql)
except:
    rows = []  # 静默
return Result.success(data=rows)  # 看起来成功，实际没数据

# ✅ 正确: 区分「真正无数据」和「查询失败」
try:
    rows = db.execute_query(sql) or []
except Exception as ex:
    logger.error(f"查询失败: {ex}")
    return Result.fail(msg=f"查询失败: {ex}")
```

**涉及接口**: `GetSalableCommodity`（依赖 `T_GOODSSALABLE`，环境中可能无数据）

## 五、通用平移检查清单

> 每次平移新接口时按此清单逐项检查。

### 5.1 参数处理
- [ ] 所有声明的参数是否真正在 SQL 或业务逻辑中使用了
- [ ] `BusinessTrade`/`shopTrade` 是否需要 `GetSubType` 展开（查 C# 源码是否有 `GetSubType` 调用）
- [ ] `BusinessTradeType` 的 SQL 映射是否正确（1→自营餐饮, 2→自营非餐饮, 3→联营）
- [ ] 多值参数（逗号分隔）是否用了 `parse_multi_ids` + `build_in_condition`

### 5.2 SQL 查询
- [ ] `BusinessTrade` 是 SQL 过滤还是内存过滤？对照 C# 确认
- [ ] JOIN 类型是否正确？自营门店场景需要 `LEFT JOIN`
- [ ] `CALCULATE_SELFSHOP` 条件是否添加（涉及 `T_PROFITCONTRIBUTE` 时）
- [ ] 查询是否包含所有需要的列（如 `SERVERPARTSHOP_NAME` 用于 fallback）

### 5.3 数据处理
- [ ] `None` / `0` / `""` 的区分是否正确（避免 `or` 吞掉 falsy 值）
- [ ] 数值格式化是否与 C# `ToString()` 一致（整数不带 `.0`）
- [ ] 利润/成本汇总是从门店列表求和还是从独立表取值
- [ ] 树形嵌套结构的汇总方向（门店→服务区→片区→根节点）

### 5.4 返回值
- [ ] 空数据时返回空列表 `[]` 还是 `None`
- [ ] 日期格式是否对齐 C#（`yyyy/M/d` 不补零）
- [ ] `Brand_ICO` URL 前缀拼接是否正确
- [ ] `MERCHANTS_ID` 加密是否使用 `des_encrypt_id`

## 七、开发环境与调试经验


### 7.2 快速差异对比模板

```python
"""C#/Python 接口差异对比脚本模板"""
import requests, json

OLD = "http://192.168.1.99:8900"   # C# 旧接口
NEW = "http://127.0.0.1:8080"      # Python 新接口

def flatten(obj, prefix=""):
    items = {}
    if isinstance(obj, dict):
        for k, v in obj.items():
            items.update(flatten(v, f"{prefix}.{k}" if prefix else k))
    elif isinstance(obj, list):
        items[f"{prefix}.length"] = len(obj)
        for i, v in enumerate(obj):
            items.update(flatten(v, f"{prefix}[{i}]"))
    else:
        items[prefix] = obj
    return items

def compare(path, params):
    r_old = requests.get(OLD + path, params=params, timeout=30).json()
    r_new = requests.get(NEW + path, params=params, timeout=30).json()
    f_old = flatten(r_old.get("Result_Data", {}))
    f_new = flatten(r_new.get("Result_Data", {}))
    all_keys = sorted(set(f_old) | set(f_new))
    diffs = [(k, f_old.get(k), f_new.get(k)) for k in all_keys if str(f_old.get(k)) != str(f_new.get(k))]
    print(f"差异数: {len(diffs)}")
    for k, vo, vn in diffs[:20]:
        print(f"  {k}: 旧={vo}, 新={vn}")

# 使用示例
compare("/CommercialApi/Revenue/GetShopSABFIList",
        {"pushProvinceCode":"340000","StatisticsMonth":"202512","ServerpartId":"416","BusinessTrade":"244"})
```

### 7.3 验证脚本的局限性

> [!IMPORTANT]
> 批量对比脚本（`compare_cached.py`）只能作为**初筛工具**，不能作为最终验收依据。

**已确认的盲区**:

| 盲区 | 说明 | 影响示例 |
|------|------|----------|
| 只比对 `DataList` | 不比对 `List` 或顶层业务字段 | `GetSalableCommodity` 的 `SalableCommodity` 字段不被检查 |
| 只比 `Result_Code` | 不深度比字段值 | 新旧都返回 `100` 但数据完全为空也会标 PASS |
| POST 样本弱 | `doc_params.json` 中部分 POST 路由参数为 `{}` | `GetMEETINGList` 基线不可靠 |
| 不检查字段缺失 | 新接口少返回几个字段不会被发现 | Examine 模块联动字段缺失 |

**正确做法**: 批量脚本初筛 → 人工精确 replay 验证 → 差异对比模板逐字段确认
