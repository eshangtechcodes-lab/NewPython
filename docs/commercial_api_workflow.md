# CommercialApi 工作流计划

> 适用范围：`CommercialApi` 已完成接口平移，但尚未完成内容级验收、问题归因、回归和上线准备。
>
> 当前目标不是继续扩展接口范围，而是把已平移接口推进到“可验收、可联调、可上线”的状态。

## 1. 适用前提

进入本工作流前，需满足以下条件：

- Python `CommercialApi` 路由已可启动并可访问
- 旧 C# `CommercialApi` 接口仍可访问，作为对照基线
- DM 数据库可访问
- 现有对比脚本可运行
- 本阶段只聚焦 `CommercialApi`，不纳入 `EShangApiMain`

## 2. 总体阶段

整体按 8 个阶段执行：

1. 建立接口台账
2. 准备测试样本
3. 开展内容级验收
4. 拆分代码问题和数据问题
5. 分批修复并回归
6. 固化自动化校验流程
7. 执行前端联调
8. 完成上线前准备

## 3. 并行执行原则

可以并行执行，但不能无约束并行。

适合并行的工作：

- 不同 Controller 的接口台账整理
- 不同 Controller 的样本准备
- 不同 Controller 的首轮内容对比
- 已完成归因的问题分批修复
- 技术要点、业务逻辑、踩坑记录整理

必须串行或设关口的工作：

- 基线样本和对比口径定义
- 公共脚本修改
- 通用中间件、公共解密、公共参数兼容逻辑修改
- 回归结果确认
- 前端联调结论确认
- 上线前检查和最终验收结论

推荐并行方式：

1. 先串行完成接口台账标准、样本口径、验收标准
2. 再按 Controller 或业务域拆成多个并行任务
3. 每个并行任务独立输出差异和归因
4. 公共问题统一合并后修复
5. 每一轮修复结束后统一回归

建议分工粒度：

- `Revenue`
- `BaseInfo`
- `Contract + Budget`
- `Examine + BusinessProcess`
- `BigData + Analysis + Customer`
- `AbnormalAudit + 低频接口`

并行约束：

- 同一时间只允许一人修改公共脚本或公共中间件
- 同一接口的归因和修复尽量由同一人闭环
- 所有并行任务必须使用同一版样本和同一版验收标准
- 每日回归前停止大范围并行修改，先收口

## 4. 阶段工作流

### 阶段 1：接口台账建立

目标：把全部 `CommercialApi` 接口纳入统一管理，明确检查范围和优先级。

执行项：

- 盘点全部 Controller、路由、请求方式
- 记录每个接口的参数类型、是否 AES、是否分页、是否树形/统计/聚合
- 标记依赖表、前端页面、业务场景
- 标记优先级：`P0`、`P1`、`P2`、`P3`

建议优先级：

- `P0`：`Revenue`、`BaseInfo`、`Contract`、`Examine`
- `P1`：`BigData`、`Customer`、`Analysis`
- `P2`：`Budget`、`AbnormalAudit`、`BusinessProcess`
- `P3`：低频接口、调试接口、联调影响小的接口

输出物：

- `CommercialApi` 接口台账
- 高风险接口清单
- 优先级列表

完成标准：

- 所有接口均已登记
- 没有“代码里存在但台账里缺失”的接口

### 阶段 2：测试样本准备

目标：为每个接口建立可重复执行的对比样本。

执行项：

- 准备空参样本
- 准备文档样本
- 准备数据库中真实有效的样本
- 为高风险接口增加边界样本：
  - 空值
  - 非法值
  - 分页边界
  - 日期边界
  - 特殊枚举值

输出物：

- 接口样本清单
- 参数样本库
- 无效样本说明

完成标准：

- 每个接口至少有 3 类样本
- 高风险接口具备边界样本

### 阶段 3：内容级验收

目标：验证 Python 接口返回内容是否与旧 C# 接口一致，而不仅是“能返回”。

检查项：

- `Result_Code`
- `Result_Desc`
- `Result_Data` 结构
- 字段名是否一致
- 字段类型是否一致
- 默认值是否一致
- 空值行为是否一致
- 默认排序是否一致
- 默认分页是否一致
- 日期格式是否一致
- 数值精度是否一致
- 树形结构和嵌套结构是否一致

专项检查：

- AES 解密类接口
- POST Body 结构
- 参数兼容行为
- 旧接口历史遗留行为
- 聚合汇总逻辑
- 明细跳转接口

输出物：

- 差异问题清单
- 差异样本
- 问题等级标记

完成标准：

- 每个接口都有验收结论：通过 / 有差异 / 阻塞
- 每个差异都可复现

### 阶段 4：问题归因

目标：把问题拆成“代码问题”和“数据问题”，避免误判。

执行项：

- 反查接口依赖表、视图、字段
- 对比 Oracle 与 DM 的数据量、字段类型、时间范围、有效 ID
- 检查以下高风险项：
  - 表缺失
  - 行数缺失
  - 字段类型不一致
  - 枚举内码不一致
  - 测试 ID 在 DM 不存在
  - 日期范围不同步
  - 旧库和新库统计口径不一致

输出物：

- 代码问题清单
- 数据问题清单
- 待同步表清单

完成标准：

- 每个差异项均有归因
- 不再把数据问题直接当作代码问题处理

### 阶段 5：问题修复与回归

目标：按优先级分批修复问题，控制返工。

修复顺序：

1. 阻塞前端联调的问题
2. 高频页面相关问题
3. 复杂统计、树形、聚合问题
4. 低频和细节问题

执行原则：

- 每次只修一批明确问题
- 每批修复后立即回归
- 回归通过后再关闭问题
- 修复记录要保留样本、原因、结果

输出物：

- 修复记录
- 回归报告
- 剩余风险清单

完成标准：

- `P0`、`P1` 问题关闭
- 高频接口全部通过验收

### 阶段 6：自动化校验固化

目标：把校验动作标准化，形成固定工作流。

执行项：

- 统一现有 `scripts` 的使用方式
- 固定批量对比命令
- 固定单接口对比命令
- 固定以下专项检查：
  - 422 扫描
  - 999 扫描
  - 超时扫描
  - 慢接口扫描
  - 数据一致性扫描
- 统一结果输出目录和命名

建议保留的回归能力：

- 全量缓存对比
- 单接口精确对比
- 差异接口复测
- 高风险接口专项校验

输出物：

- 回归命令清单
- 结果目录规范
- 每日回归模板

完成标准：

- 修复后可快速重复回归
- 回归结果可追踪、可复盘

### 阶段 7：前端联调

目标：从“接口正确”推进到“业务流程可用”。

执行项：

- 按页面和业务流程进行联调
- 检查列表、筛选、分页、明细、导出、图表、空态、异常态
- 补充记录页面侧暴露出的业务问题

输出物：

- 联调问题单
- 页面通过清单
- 页面阻塞清单

完成标准：

- 主流程页面联调通过
- 无阻塞上线的问题遗留

### 阶段 8：上线前准备

目标：让 `CommercialApi` 具备上线条件。

执行项：

- 排查慢 SQL 和超时接口
- 检查日志、异常码、健康检查
- 整理配置项、环境变量、启动方式
- 准备部署说明和回滚方案
- 明确试运行观察指标

输出物：

- 上线检查清单
- 性能问题清单
- 部署说明
- 回滚方案

完成标准：

- 上线门槛明确
- 试运行观察项明确

## 5. 知识沉淀要求

整个执行过程中，必须同步沉淀以下信息，供后续 AI 和新成员直接阅读。

必须记录的内容：

- 技术要点
- 业务逻辑
- 参数含义和特殊规则
- 数据表依赖关系
- 已知坑和修复方法
- 差异归因结论
- 回归方法和验证口径

建议记录位置：

- `workLog/YYYY-MM-DD.md`
  - 记录每天做了什么、发现了什么、修了什么、剩下什么
- `docs/commercial_api_ai_context.md`
  - 记录稳定结论，给 AI 做长期上下文
- `docs/api_migration_pitfalls.md`
  - 记录通用坑、重复出现的问题、修复规则

记录原则：

- 日志记录“过程”
- AI 上下文记录“稳定结论”
- 踩坑文档记录“可复用规则”

每完成一个接口或一类问题，至少补充以下内容：

- 接口用途
- 核心业务逻辑
- 关键 SQL 或数据来源
- 参数特殊含义
- 与旧接口不一致的原因
- 修复方法
- 回归方式

供 AI 阅读时，优先保证以下特点：

- 用固定标题
- 一条结论只表达一个点
- 结论和样例分开写
- 标明适用接口和日期
- 标明该结论是否已验证

## 6. 每日执行节奏

建议按日推进：

1. 上午确定当天接口范围和问题范围
2. 白天执行对比、归因、修复
3. 下班前跑一次回归
4. 更新差异状态、风险项、次日计划

建议每两天做一次小结，内容包括：

- 已验收接口数
- 新发现问题数
- 已关闭问题数
- 数据问题与代码问题占比
- 当前阻塞项

### 里程碑 1：台账完成

- 所有 `CommercialApi` 接口已登记
- 样本准备方案已明确

### 里程碑 2：首轮验收完成

标志：

- 全部接口至少完成一轮内容对比
- 已形成完整差异清单

### 里程碑 3：高优问题关闭

标志：

- `P0`、`P1` 问题已关闭
- 高频页面相关接口已回归通过

### 里程碑 4：联调通过

标志：

- 前端主流程联调通过
- 无阻塞性业务问题

### 里程碑 5：具备上线条件

标志：

- 回归流程固化
- 性能问题已处理或已评估
- 部署和回滚方案已就绪

## 8. 平移完成后的下一步

`CommercialApi` 平移完成后，正式进入以下顺序：

1. 内容级验收
2. 数据一致性校验
3. 问题修复与回归
4. 前端联调
5. 性能优化
6. 上线准备
7. 试运行观察

## 9. 推荐执行顺序

如果需要按短周期推进，建议按以下顺序执行：

1. 建立接口台账
2. 准备测试样本
3. 优先核对 `P0` 接口
4. 输出差异清单并归因
5. 修复 `P0`、`P1` 问题并回归
6. 完成前端联调
7. 做性能和上线准备

## 10. 风险提示

本阶段最容易出现的偏差有：

- 只看接口是否返回，未检查内容一致性
- 把数据问题误当成代码问题
- 高风险接口没有边界样本
- 修复后未做批量回归
- 前端联调前就判断“已完成”

因此，验收阶段必须坚持以下原则：

- 先建台账，再做对比
- 先做归因，再做修复
- 每修必回归
- 以前端业务流程通过作为最终验收条件
